# Plan: Add `--continue` Flag for Resuming Last Playback

## Goal

Add a `--continue` flag to the YouTube Music CLI that resumes the last played playlist or song.

## Current State Analysis

### Player State Persistence

The `player-state.service.ts` already persists:

- `currentTrack` - the last played track
- `queue` - the entire queue
- `queuePosition` - position in the queue
- `progress` - time position in current track
- `volume`, `shuffle`, `repeat`

### Existing Flag Patterns

Flags are defined in:

- `source/cli.tsx` - meow configuration (lines 90-137)
- `source/types/cli.types.ts` - Flags interface

Flags are processed in:

- `source/main.tsx` - `Initializer` component for TUI mode
- `source/main.tsx` - `HeadlessLayout` for headless mode

## Implementation Plan

### Step 1: Add `continue` to Flags interface

**File:** `source/types/cli.types.ts`

- Add `continue?: boolean` to the Flags interface

### Step 2: Add `--continue` flag to meow config

**File:** `source/cli.tsx`

- Add flag definition:
  ```typescript
  continue: {
    type: 'boolean',
    default: false,
  },
  ```
- Add help text to usage section: `--continue, -c          Resume last played track/playlist`

### Step 3: Update flag processing

#### TUI Mode (Initializer in main.tsx)

Add handling for `flags?.continue`:

- Load persisted player state
- If there's a saved track, restore queue and queue position
- Resume playback from saved position (or start from saved queue position)

#### Headless Mode (HeadlessLayout in main.tsx)

Add handling for `flags?.continue`:

- Load persisted player state
- Play from saved position
- Output "Resuming: {track title}"

### Step 4: Handle "no saved state" case

- If `--continue` is used but no saved state exists:
  - In TUI mode: Show notification "No previous playback to resume"
  - In headless mode: Output error and exit with code 1

## Questions for Clarification

1. **Behavior**: Should `--continue` resume from the exact time position (progress), or just restart the last track/playlist from the beginning?
   - _Recommendation_: Resume from saved queue position, but restart from beginning of track (not time position) to ensure clean playback

2. **Queue restoration**: Should the entire queue be restored, or just the current track?
   - _Recommendation_: Restore entire queue for full "continue" experience

3. **Priority**: If both `--continue` and another playback flag (e.g., `--playTrack`) are provided, which takes precedence?
   - _Recommendation_: `--continue` should not combine with other playback flags; if both are provided, show warning and use `--continue` only
